# A2A Protocol: Complete Reference Guide

## Introduction to Agent-to-Agent (A2A) Protocol

The Agent-to-Agent (A2A) protocol is an open standard that enables different AI agents to securely communicate, collaborate, and solve complex problems together. It provides a standardized way for agents from different providers or built using different frameworks to interoperate seamlessly.

### Core Principles

1. **Interoperability**: Bridge the communication gap between disparate agentic systems
2. **Collaboration**: Enable agents to delegate tasks, exchange context, and work together
3. **Discovery**: Allow agents to dynamically find and understand capabilities of other agents
4. **Flexibility**: Support various interaction modes (synchronous, streaming, asynchronous)
5. **Security**: Facilitate secure communication using standard web security practices

## Key Concepts

### Agent Card

The Agent Card is a JSON metadata document published by an A2A Server, describing:
- Agent identity (name, description, version)
- Capabilities (streaming, push notifications, state history)
- Skills (specific functions the agent can perform)
- Service endpoint URL
- Authentication requirements

Agent Cards act as the agent's "business card" or "resume", enabling discovery and capability negotiation.

**Example Agent Card Structure:**
```json
{
  "name": "Weather Agent",
  "description": "Provides weather information and forecasts",
  "version": "1.0.0",
  "url": "https://api.example.com/weather-agent",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    {
      "id": "current_weather",
      "name": "Current Weather",
      "description": "Get current weather for a location",
      "inputModes": ["text/plain"],
      "outputModes": ["application/json"]
    }
  ]
}
```

The Agent Card is typically served at: `/.well-known/agent-card.json`

### Task Management

A **Task** is the fundamental unit of work in A2A protocol. Each task:
- Has a unique identifier (taskId)
- Progresses through a defined lifecycle
- Is stateful (tracks progress over time)
- Can be completed immediately or over multiple exchanges

**Task Lifecycle States:**
1. **SUBMITTED**: Task received but not yet processed
2. **WORKING**: Task is actively being processed
3. **INPUT_REQUIRED**: Agent needs additional input from client
4. **COMPLETED**: Task finished successfully
5. **FAILED**: Task failed with an error
6. **CANCELED**: Task was canceled

### Messages and Parts

Communication in A2A happens through **Messages**. Each message:
- Has a role: "user" (from client) or "agent" (from server)
- Contains one or more **Parts**
- Parts are the smallest unit of content (text, file, data, etc.)

**Message Structure:**
```json
{
  "role": "user",
  "parts": [
    {
      "type": "text",
      "content": "What's the weather in San Francisco?",
      "mimeType": "text/plain"
    }
  ]
}
```

### Artifacts

An **Artifact** is an output generated by the agent as a result of a task. Examples:
- Generated documents
- Images
- Structured data
- Analysis results

## A2A Communication Flow

### Basic Workflow

1. **Discovery Phase**
   - Client fetches Agent Card from `/.well-known/agent-card.json`
   - Client learns about agent's capabilities and skills
   - Client determines if agent can fulfill the needed task

2. **Task Submission**
   - Client generates unique taskId
   - Client sends initial message with taskId to agent
   - Agent creates task and begins processing

3. **Processing**
   - Agent processes the request
   - May require multiple message exchanges
   - Task status updates as processing progresses

4. **Completion**
   - Task reaches terminal state (completed/failed/canceled)
   - Agent returns final response and artifacts
   - Client can retrieve task history

### Example: Multi-Agent Collaboration

```
User -> Host Agent: "Find and book a restaurant in Tokyo"
    |
    v
Host Agent -> Restaurant Search Agent: "Find restaurants in Tokyo"
    |
    v
Restaurant Search Agent -> Host Agent: [List of restaurants]
    |
    v
Host Agent -> Booking Agent: "Book table at Restaurant X"
    |
    v
Booking Agent -> Host Agent: "Booking confirmed"
    |
    v
Host Agent -> User: "Booked table at Restaurant X for 7pm"
```

## Agent Skills

Skills describe specific capabilities an agent offers. Each skill includes:

- **id**: Unique identifier
- **name**: Human-readable name
- **description**: What the skill does
- **tags**: Keywords for discovery
- **examples**: Sample queries
- **inputModes**: Accepted input formats
- **outputModes**: Output formats

**Example Skill Definition:**
```json
{
  "id": "translate_text",
  "name": "Text Translation",
  "description": "Translate text between languages",
  "tags": ["translation", "nlp", "languages"],
  "examples": [
    "Translate 'hello' to Spanish",
    "Convert this text to French"
  ],
  "inputModes": ["text/plain"],
  "outputModes": ["text/plain", "application/json"]
}
```

## Authentication

A2A supports standard authentication schemes aligned with OpenAPI specification:

- **API Keys**: Simple token-based authentication
- **OAuth 2.0**: Delegated authorization
- **OpenID Connect**: Identity layer on OAuth 2.0
- **Bearer Tokens**: Token passed in Authorization header

Authentication is declared in the Agent Card and enforced by the agent server.

## Transport Protocols

A2A primarily uses HTTP as its transport layer, with support for:

1. **Request/Response**: Standard synchronous HTTP requests
2. **Server-Sent Events (SSE)**: Real-time streaming updates
3. **Push Notifications**: Asynchronous updates to client endpoints

Each agent declares supported transports in its Agent Card.

## Implementation Best Practices

### For Agent Servers

1. **Publish a Complete Agent Card**
   - Include all skills and capabilities
   - Keep documentation URL current
   - Update version on breaking changes

2. **Implement Proper Task Management**
   - Generate unique task IDs
   - Track task state accurately
   - Clean up completed tasks after retention period

3. **Handle Errors Gracefully**
   - Return meaningful error messages
   - Use appropriate HTTP status codes
   - Include error details in task status

4. **Support Discovery**
   - Make Agent Card accessible without authentication
   - Use standard path: `/.well-known/agent-card.json`
   - Keep card schema-compliant

### For Agent Clients

1. **Always Fetch Agent Card First**
   - Understand agent capabilities before sending tasks
   - Respect declared input/output modes
   - Check authentication requirements

2. **Generate Unique Task IDs**
   - Use UUIDs or similar guaranteed-unique identifiers
   - Include in all task-related requests

3. **Handle Async Tasks Properly**
   - Poll for status updates if needed
   - Implement timeout handling
   - Support push notifications if available

4. **Maintain Context**
   - Include sessionId for related tasks
   - Reference previous messages when relevant
   - Build conversation history

## Security Considerations

1. **Credential Management**
   - Keep API keys and tokens secure
   - Use environment variables, not hardcoded values
   - Rotate credentials regularly

2. **Input Validation**
   - Validate all incoming messages
   - Sanitize user inputs
   - Implement rate limiting

3. **Access Control**
   - Implement proper authorization
   - Verify client permissions for each task
   - Log all access attempts

4. **Data Privacy**
   - Don't log sensitive information
   - Encrypt data in transit (HTTPS)
   - Comply with privacy regulations

## A2A vs Other Protocols

### A2A vs MCP (Model Context Protocol)

- **A2A**: Agent-to-agent communication, task delegation, inter-agent collaboration
- **MCP**: Agent-to-tool communication, resource access, data retrieval

These protocols complement each other:
- Use MCP for agents to access tools (databases, APIs, files)
- Use A2A for agents to communicate with each other

### A2A vs REST APIs

- **REST**: Service-to-service, stateless, resource-oriented
- **A2A**: Agent-to-agent, stateful (tasks), capability-oriented

A2A builds on REST/HTTP but adds:
- Standardized discovery (Agent Cards)
- Task lifecycle management
- Multi-turn conversations
- Agent-specific conventions
